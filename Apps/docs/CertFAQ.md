# <a name="microsoft-365-certification-faq"></a>Microsoft 365认证常见问题解答

## <a name="introduction"></a>简介
以下是 ISV (独立软件) 开始认证时 (常见问题) 常见问题Microsoft 365常见问题。  如果此处未涵盖你拥有的任何查询，请通过Microsoft 365联系应用认证 AppCert@Microsoft.com。
本文档面向 ISV，有关 Microsoft 365 安全与合规计划的一般信息可在"Microsoft 365[合规性计划"页下找到](https://docs.microsoft.com/microsoft-365-app-certification/docs/enterprise-app-certification-guide)。 


|问题                                       |扩展定义                          |
|----------------------------------------       |---------------------------------------------|
|我尚未针对行业认可的框架（如 PCI DSS、SOC 2 或 ISO 27001）通过审核。  这是否意味着我无法申请Microsoft 365认证？ | 不，获得这些行业认可的框架之一不是获得认证Microsoft 365要求。|
|我已针对行业认可的框架通过了外部审核。 此计数能否计入Microsoft 365认证？ | 简短回答是"是"。  目前，Microsoft 365认证规范接受 PCI DSS、SOC 2 和 ISO27001 外部框架的证据。 认证Microsoft 365指南已映射这些现有外部框架的对应位置;但是，我们发现在某些情况下，现有标准/框架没有完全一致。 因此，Microsoft 365认证团队将审查所提供的标准/框架证据，从而标记符合Microsoft 365认证中的哪些控制措施。 |
|如果尚未进行外部 GDPR 评估，如何证明 GDPR 合规性？ | Microsoft 不要求独立审查 Microsoft 365 认证 GDPR 一致性，因为这是一种/或方案，我们将接受自证明，我们可以在未发生外部审查时独立验证。 由于这不仅仅是一项评估，并且对于我们在我们的过程中要收集的证据，审查隐私策略和内部流程是我们实现 GDPR 控制的方式。 出于我们在 GDPR 控制中寻找的目的，它主要涉及审查隐私策略以确保它们满足基本 GDPR 要求;例如，正在处理哪些个人数据、处理是否合法、指出数据主体权利、用户如何执行主题访问请求 (SAR) 、ISV 如何执行 SAR、ISV 公司详细信息和数据保留详细信息。|
|我们已经历渗透测试;但是，我们还没有进行"干净"渗透测试，因为我们尚未进行渗透重新测试。  我们是否需要进行重新测试并拥有一份干净的报告？ | 该Microsoft 365认证规范不要求 ISV 进行渗透测试重新测试，从而提供足够的修正证据来证明渗透测试报告中发现的问题已修复。|
|请求的一些文档和证据很敏感，是否有保密协议 (NDA) 已到位？ | 是的，你提交的一些信息是公共信息，有些可能是机密信息。  如果你已有 Microsoft 的 NDA，该 NDA 的条款将适用于你提交的机密信息。  如果你没有 Microsoft 的 NDA，你在合作伙伴中心Publisher协议保密条款将适用于该机密信息。|
|作为认证评估的一部分，我们如何安全地传输Microsoft 365和证据？ | 目前，Microsoft 没有安全共享此信息的平台。 建议你通过已有的任何安全机制来共享此信息。 许多 ISV 将利用OneDrive并共享到认证团队的Microsoft 365链接。|
|我们刚刚实施了一些额外的安全流程，以满足某些 Microsoft 365 认证控制措施，这是否意味着我们需要等待 12 个月才能通过认证？|否，Microsoft 确认你可能需要开发其他安全流程来弥补现有安全流程与认证预期Microsoft 365差距。 认证Microsoft 365将审阅新开发的文档过程，并检查至少已执行一次该过程的证据。 不需要其他历史证据，因为此证据不适用于这些新开发的过程。 12 个月后，将在年度评估期间开始评估历史证据的样本。|
|我负责提供哪些服务？| 在评估过程中，认证分析师将审阅提供的文档和证据，以评估你是否符合Microsoft 365控制措施。 作为工作的一部分，Microsoft 365认证团队将请求包含体系结构详细信息、图表、数据存储详细信息、应用设计详细信息、策略和流程文档、配置文件和屏幕截图的信息。 在某些情况下，或者如果你更容易，可以安排屏幕共享会话以显示认证分析员的证据。 如果要使用现有合规性框架来支持评估活动，需要有足够的文档来演示外部审核员/评估员已评估并确认已就位的情况。 如果支持文档无法提供必要的叙述来准确演示如何满足外部安全框架中的控制措施，Microsoft 365 认证团队将无法利用外部安全框架来支持 Microsoft 365 认证评估。|
|获得认证是否需要我更改我的当前基础结构？| 不太可能需要对基础结构进行重大更改才能获得 Microsoft 365 认证。 这些控件基于行业安全最佳做法，很可能已实现。  在大多数情况下，我们已经看到;ISV 必须更新内部流程，以弥补当前工作实践与认证Microsoft 365差距。 如果这是一个问题，Microsoft 建议你查看最新的 Microsoft 365 认证控制，可在 Microsoft 365 认证提交指南上找到这些控件，以确保当前部署的环境和工作实践符合定义的控制措施。|
|Microsoft 是否对应该用于满足认证要求的特定组件/基础结构/软件提出了建议？| Microsoft 不提供有关满足认证控制要求的解决方案Microsoft 365建议。 可以使用任何商业或开源产品/服务，因为它们受到主动支持和维护。|
|完成评估需要多久？| 通常，评估平均可能需要 30 天才能完成，但这取决于许多变量。 完成时间长短取决于：用于支持应用/加载项的托管环境的大小、支持应用/加载项的托管环境类型，以及 ISV 响应证据请求的提示程度。|
|我需要为此过程分配多少时间？ | 大多数工作只是及时收集文档和证据。 此后，每周不应超过几小时，即完成评估过程。 一些变量会影响所需时间：环境的大小将影响收集请求的证据所需的时间量，以及是否有可用于支持评估的任何外部安全框架。  如果已制定外部安全框架，并且可以提供足够的支持文档，则认证分析师可以使用这些外部评估满足 Microsoft 365 控制措施的子集，而无需提供其他证据。|
|为什么评估有固定的 60 天时间范围？ | 我们已对执行评估的时间长度设置了限制，因为已收集的证据可能会过期，评估所花的时间越长。  这是一个时间点评估，因此需要为完成分配一个合适的时间段。 提交初始文档提交后，我们将响应证据请求。 60 天期限从您收到证据请求时开始。 应[Microsoft 365](https://docs.microsoft.com/microsoft-365-app-certification/docs/certification-submission-guide)认证提交指南，并且你应确信在提交初始证据提交之前，所有控制措施均能得到满足。|
|如果在 60 天内评估未完成，会发生什么情况？ | 遗憾的是，如果在 60 天的时间范围内未完成评估，Microsoft 将针对评估标记为失败。 此标记仅适用于内部统计信息，永远不会发布。 你将能够立即重新启动评估过程，但是将请求你重新发送新证据以支持新应用程序。|
|认证需要我Microsoft 365多少？ | 目前，你可以免费完成Microsoft 365认证。 |
|此计划下渗透测试的成本是多少？ | 如果你的应用必须接受渗透测试（其中这不是安全活动的一部分），则渗透测试可以在"Microsoft 365认证"下完成，并且是免费的。 渗透测试的范围仅限于应用和支持基础结构，这些基础结构在认证Microsoft 365范围内。 |
|你是否具有可用于播发我们的应用已经过认证的营销材料？ | 我们目前不提供促销或营销材料。 尽管我们鼓励你告诉客户你的应用已收到 Microsoft 365 认证，但唯一可用的图形是 AppSource 和 Microsoft Docs 页面中的应用一览上显示的图形。 如果有任何市场营销材料可用，我们将更新本文档。
|执行评估时要查找哪些级别的证据？ | 认证评估Microsoft 365提供的证据必须能够提供足够保证，确保你满足正在评估Microsoft 365认证控制措施的特定要求。  证据的形式可以是配置文件、设置或证据的屏幕截图、策略/过程文档或屏幕共享会话，以向认证分析师演示证据。  下面是两个示例：|
||**评估** 活动："演示防病毒软件正在所有采样的系统组件中运行。" – 对于此控件，你可以提供示例中支持防病毒的每台设备的屏幕截图（显示运行的防病毒过程）。或者，如果你有用于防病毒的集中管理控制台，则也许可以从该管理控制台进行演示。|
||**评估活动**："演示如何识别新的安全漏洞。" – 此控件来自"修补程序管理"部分。 目的是，您具有一个正式记录的流程，用于确定新的安全漏洞。  这可能位于源代码内，但也需要位于支持环境中，例如 Windows 漏洞、Web 依赖项中的漏洞 (例如 AngularJS、JQuery 等 ) 。  您应该有一个记录的流程，用于识别新的安全漏洞，以便提供记录的进程文档。  除了文档之外，还需要提供正在遵循此过程的证据;例如，如果你正在利用 npm 审核等功能来检查漏洞的依赖性，那么提供报告示例将提供证据。  如果您利用多个进程（即，对于不同的系统组件），则需要提供所有过程的证据。|
